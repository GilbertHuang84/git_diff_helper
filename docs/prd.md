# 产品需求文档 (PRD): GD (Git-Diff-Helper)

## 1. 产品定位

**GD** 是一款面向开发者和 Tech Lead 的命令行工具。它通过 GitLab API 快速审计多个私有代码仓的分支差异，并提供一键创建同步 MR 的功能，解决微服务架构下代码同步状态难追踪、发布流程繁琐的问题。

## 2. 核心功能需求

### 2.1 配置管理与继承

* **全局配置**：存储于 `~/.gd/config.yaml`，包含 `gitlab_url`、`token` 以及默认分支（`master`/`develop`）。
* **分组管理**：支持将仓库（Repo）按业务模块分组（Group），支持对组进行批量操作。
* **继承规则**：分支定义遵循 `Repo > Group > Global` 的优先级覆盖逻辑。

### 2.2 仓库与状态扫描

* **仓库发现**：支持通过关键字搜索远端 GitLab 仓库名及 ID。
* **完整性检查 (`verify`)**：验证本地配置的仓库在远端是否真实存在、权限是否正常。
* **状态快照 (`status`)**：
* **Test vs Prod**：识别开发分支领先（Ahead）或落后（Behind）主干的情况。
* **Tag vs Prod**：识别最新发布版本与主干代码的偏移。


* **可视化反馈**：使用 🟢/🔵/🔴 颜色标识同步、领先、落后等风险状态。

### 2.3 自动化流水线操作 (`sync`)

* **自动创建 MR**：
* 检查是否存在存量未关闭 MR，若有则直接返回链接。
* 若无 MR 且存在差异，则自动创建 `develop -> master` 或 `master -> develop` 的 MR。
* 将 Commit 摘要自动填充至 MR 描述。


* **打 Tag 功能**：在代码合入后，支持通过命令快速给主干打上遵循语义化版本的最新 Tag。

## 3. 交互逻辑与命令规范

### 3.1 基础管理

| 命令 | 说明 |
| --- | --- |
| `gd search <key>` | 搜索远端所有可访问仓库及 ID |
| `gd add <id> [-g group]` | 将仓库加入本地配置，可选指定分组 |
| `gd list` | 查看当前配置的所有分组及仓库架构 |

### 3.2 审计与状态

| 命令 | 说明 |
| --- | --- |
| `gd verify` | 检查配置仓库的权限和分支有效性 |
| `gd status` | 全局总览：所有仓库的 Diff 简报（颜色标记） |
| `gd status <id>` | 交互式详情：查看特定仓的 Commit 列表、作者及内容 |

### 3.3 同步操作

| 命令 | 说明 |
| --- | --- |
| `gd sync <id> --to-master` | 创建开发到主干的 MR（发布预览） |
| `gd sync <id> --to-dev` | 创建主干到开发的 MR（回流同步） |
| `gd tag <id> <version>` | 在指定仓库打上新 Tag |

## 4. 开发过程的操作路径 (Implementation Path)

作为开发者，建议按以下阶段进行迭代开发：

### 第一阶段：基础架构与配置引擎 (Core)

1. **环境初始化**：实现 `~/.gd/` 目录的自动创建和 `config.yaml` 的读写逻辑。
2. **继承逻辑实现**：编写配置解析器，确保 Repo 能够正确继承 Group 或 Global 的分支配置。
3. **GitLab 客户端封装**：基于 `python-gitlab` 或原生 `requests` 封装基础 API（Projects, Commits, Compare）。

### 第二阶段：只读审计功能 (Audit)

1. **实现 `gd search**`：确保能从远端拉取项目列表。
2. **实现 `gd status` (核心难点)**：
* 引入 **异步并发 (Asyncio)** 调用 API，提升多仓扫描速度。
* 编写逻辑对比两个 Ref 的 Commit 差异，并进行颜色高亮处理。


3. **实现交互式详情**：利用 `Rich` 库渲染表格，支持用户查看具体 Commit。

### 第三阶段：写操作与 MR 自动化 (Automation)

1. **MR 逻辑开发**：编写“先查存量 MR，再查差异，最后创建”的闭环逻辑。
2. **权限容错**：处理 API 返回的 403（权限不足）、409（MR 已存在）等异常状态。
3. **Tag 模块**：实现获取最新 Tag 并生成下一个建议版本号的功能。

### 第四阶段：优化与分发 (Polish)

1. **本地缓存**：在 `~/.gd/cache` 记录最后一次扫描的 Commit ID，减少不必要的对比请求。
2. **打包分发**：使用 `PyInstaller` 打包成独立二进制文件或通过 `pip` 安装，确保随处可用。

## 5. 关键风险点评估

* **GitLab API 限制**：频繁调用 Compare 接口可能导致私有实例负载过高。*对策：加入请求频率限制（Rate Limiting）和缓存。*
* **MR 冲突处理**：API 创建 MR 后可能无法合并。*对策：在 CLI 输出中明确标记 `Mergeable Status`，并附带网页链接引导用户手动解决。*
* **隐私安全**：Token 存储在本地。*对策：建议权限设定为 600 (仅当前用户读写)，并支持从系统环境变量读取敏感信息。*
